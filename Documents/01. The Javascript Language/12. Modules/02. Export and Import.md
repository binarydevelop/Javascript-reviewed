## Export and Import
We can label any declaration as exported by placing export before it, be it a variable, function or a class.

**No semicolons after export class/function**

### Export Default 
In practice, there are mainly two kinds of modules.

Modules that contain a library, pack of functions, like ``say.js`` above.
Modules that declare a single entity, e.g. a module ``user.js`` exports only ``class User``.

Mostly, the second approach is preferred, so that every â€œthingâ€ resides in its own module.

Modules provide a special ``export default`` (â€œthe default exportâ€) syntax to make the â€œone thing per moduleâ€ way look better.

Put export default before the entity to export:
```js

  constructor(name) {
    this.name = name;
  }
}
```

â€¦And then import it without curly braces:
```js
// ğŸ“ main.js

new User('John');
```
``Import`` needs curly braces for _named exports_ and doesnâ€™t need them for the _default_ one.

**Named export**	               **Default export**
``export class User {...}``         ``export default class User {...}``
``import {User} from ...``         ``import User from ...``

As there may be at most one default export per file, the exported entity may have no name.

For instance, these are all perfectly valid default exports:
```js
export default class { // no class name
  constructor() { ... }
}
-----
export default function(user) { // no function name
  alert(`Hello, ${user}!`);
}
```
Not giving a name is fine, because there is only one export default per file, so import without curly braces knows what to import.

### The â€œdefaultâ€ name
In some situations the ``default`` keyword is used to reference the ``default`` export.

For example, to export a function separately from its definition:
```js

}

```

Or, another situation, letâ€™s say a module user.js exports one main â€œdefaultâ€ thing, and a few named ones (rarely the case, but it happens):

Hereâ€™s how to import the default export along with a named one:

And, finally, if importing everything ``*`` as an object, then the default property is exactly the default export:

- Named exports are explicit. They exactly name what they import, so we have that information from them; thatâ€™s a good thing.

Named exports force us to use exactly the right name to import:
```js

// import {MyUser} won't work, the name must be {User}
â€¦While for a default export, we always choose the name when importing:

// could be import Anything... and it'll still work
```


### Rexport
â€œRe-exportâ€ syntax export ... from ... allows to import things and immediately export them (possibly under another name), like this:
```js

```

**Why would that be needed? Letâ€™s see a practical use case.**
Imagine, weâ€™re writing a â€œpackageâ€: a folder with a lot of modules, with some of the functionality exported outside, and many modules are just â€œhelpersâ€, for internal use in other package modules.
The file structure could be like this:
```

```

Weâ€™d like to expose the package functionality via a single entry point. In other words, a person who would like to use our package, should import only from the â€œmain fileâ€ ``auth/index.js``.
Like this:
```js

```
The â€œmain fileâ€, ``auth/index.js`` exports all the functionality that weâ€™d like to provide in our package.

The idea is that outsiders, other programmers who use our package, should not meddle with its internal structure, search for files inside our package folder. We export only whatâ€™s necessary in ``auth/index.js`` and keep the rest hidden from prying eyes.

As the actual exported functionality is scattered among the package, we can import it into auth/index.js and export from it:
```js

// import default as User and export it

export {User};
...
```

Now users of our package can import {login} from "auth/index.js".

The syntax export ... from ... is just a shorter notation for such import-export:
```js

// re-export login/logout

// re-export the default export as User

...
```
The notable difference of ``export ...`` from compared to ``import/export`` is that re-exported modules arenâ€™t available in the current file. So inside the above example of ``auth/index.js`` we canâ€™t use re-exported ``login/logout ``functions.

### Re-exporting the default export
The default export needs separate handling when re-exporting.

Letâ€™s say we have ``user.js`` with the export ``default class User`` and would like to re-export it:
```js

  // ...
}
```
We can come across two problems with it:

- ``export User from './user.js'`` wonâ€™t work. That would lead to a syntax error.

To re-export the default export, we have to write ``export {default as User}``, as in the example above.

- ``export * from './user.js'`` re-exports only named exports, but ignores the default one.

If weâ€™d like to re-export both named and default exports, then two statements are needed:
```js

export {default} from './user.js'; // to re-export the default export
```
